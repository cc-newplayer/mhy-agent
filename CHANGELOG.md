# B站舆情分析系统 - 开发日志

## 项目概述

针对B站视频评论区的自动化舆情分析流水线，覆盖数据采集、情感分类、负面深度分析、运营建议生成全链路。

当前样本：《崩坏：星穹铁道》千冶•刃角色PV（BV1PnV46DEP4）

---

## 技术栈

- Python 3.13
- Playwright（浏览器自动化，绕过B站风控）
- Claude API（claude-opus-4-7，情感分类 + 建议生成）
- jieba（中文分词）
- matplotlib（图表绘制）

## 变更记录

### 2026-06-02 v1.0 初版搭建

**评论爬取**

1. 初始方案：`requests` + B站API → 被风控，未登录只返回3条
2. 尝试带Cookie的requests → 仍然只有3条（缺浏览器指纹/wbi签名）
3. 最终方案：Playwright浏览器自动化，通过`page.goto(API_URL)`导航获取JSON
4. 接口选择：从旧接口`/x/v2/reply`（页码翻页，受限）切换到新接口`/x/v2/reply/main`（游标翻页，完整返回）
5. 增加楼中楼抓取：对有子回复的主评论调用`/x/v2/reply/reply`获取子回复（每条最多3页60条）
6. 数据量：100条主评论 + 1285条子回复 = 1385条

**情感分析**

1. 调用Claude API（claude-opus-4-7），每10条一批，批间限流2秒
2. 三分类：正面/负面/中性
3. 结果：正面48.4% / 中性36.1% / 负面15.5%

**负面深度分析**

1. jieba分词 + 自定义停用词表
2. 去除"回复 @xxx:"噪声前缀
3. 补充停用词：回复、就是、不是 等虚词
4. Top 10 负面高频词：sp、角色、待遇、剧情、pv、宣发、主推 等
5. Claude自动生成3条运营建议

**踩过的坑**

| 问题 | 原因 | 解决 |
|------|------|------|
| requests被B站风控 | 缺浏览器指纹/wbi签名 | 改用Playwright |
| Playwright fetch()返回-352 | B站对XHR额外校验 | 改用page.goto()导航 |
| 中转站模型503 | claude-sonnet-4通道被占满 | 换用claude-opus-4-7 |
| 高频词全是噪声 | "回复""就是"等虚词干扰 | 补充停用词+正则清洗 |
| 只有100条样本，负面占比低 | 未抓楼中楼 | 增加子回复抓取，样本1385条 |

### 2026-06-02 v2.0 负面分析重构（混合方案C）

**问题**

v1的jieba纯词频统计存在两个硬伤：
1. 高频词有大量废词（"可恶""几把""时候"），不携带有效信息
2. 关键词与上下文脱节——看到"待遇"不知道指谁的待遇

**方案：TF-IDF + Claude语义归类**

1. TF-IDF提取候选词：以全量评论为语料库，计算负面评论中区分度高的词（非简单词频）
2. Claude归纳议题：把候选词+典型评论喂给Claude，让它归纳为5-8个具体"负面议题"，每个议题有标签+描述+关联词
3. Claude分类：对每条负面评论标注所属议题（可多选）
4. 按议题聚合典型评论：每个议题下按综合评分（likes×2 + 主评论加权50）排序取top 3

**效果对比**

| | v1 | v2 |
|--|-----|-----|
| 输出 | "sp(32)、角色(30)、待遇(25)" | "宣发资源不足(37条): 刃相比其他角色获得的宣发资源明显偏少" |
| 典型评论 | 全局top5负面评论 | 每个议题各3条，按点赞+位置排序 |
| 建议 | 3条通用建议 | 每个议题对应1条具体建议 |

**新增依赖**: scikit-learn（TF-IDF）

**踩坑**: 子回复中"回复 @xxx:"前缀会干扰分词，需正则清洗

**辅助脚本**

1. 新增 `login.py`：自动轮询检测B站登录状态，无需手动按回车

**踩过的坑**

| 问题 | 原因 | 解决 |
|------|------|------|
| agent导入其他脚本时stdout崩溃 | 多次TextIOWrapper wrap关闭buffer | 加encoding检查guard |

| .env读取第一个key带乱码 | PowerShell Out-File加了UTF-8 BOM | 用WriteAllText无BOM写入 |
| 前端发消息无响应 | 500错误未被前端捕获展示 | 加response.ok检查 |

### 2026-06-04 v3.1 新增文件读取工具 + 报告命名规范

**问题发现**

使用 Agent 时发现：同一对话内跑完分析后可以追问"梳理议题一的讨论脉络"（Agent靠对话历史中的工具返回数据推理），但新开对话时无法做到——Agent 没有工具读取已有报告，缺乏原材料无法推理。且目录下有多份报告时，Agent 无法区分该读哪份。

**改动**

1. 新增 `list_files` 工具：列出项目目录下所有分析报告和数据文件，Agent 可自主查看有哪些可用的历史结果
2. 新增 `read_file` 工具：读取指定文件内容（.json/.md/.csv），JSON 大文件自动截断前50条，文本文件截断8000字符
3. 报告文件命名加入 BV号+日期：`negative_report_BV1xxx_20260604.md`，Agent 通过文件名即可判断对应哪个视频
4. 图表文件同步改名：`negative_chart_BV1xxx_20260604.png`

**效果**

- 新对话中用户说"帮我看看上次BV1xxx的分析"→ Agent 调 list_files 找到文件 → 调 read_file 读取 → 基于内容推理回答
- 多份报告共存时，Agent 通过文件名中的BV号和日期自主判断，不确定时会反问用户

**踩坑**

| 问题 | 原因 | 解决 |
|------|------|------|
| list_files 工具无响应，agent 死循环 | 中转站对无参数工具返回空content（stop_reason=tool_use但content=[]） | 给工具加required参数 + agent loop加空结果防御 |

### 2026-06-04 v3.2 情感分析准确性评估

**方法**

从 BV1PnV46DEP4（星穹铁道·刃PV）的情感分析结果中等比抽样150条（正面/中性/负面各50），人工标注后与模型输出对比。

**结果**

- 总准确率：**63.3%**（95/150）
- 负面识别最好：Recall 85%，F1 75.6%
- 正面误判严重：Precision 仅 50%，大量中性评论被错标为正面
- 中性被侵蚀最多：23条被标为正面，9条被标为负面

**混淆矩阵**

|  | 模型→正面 | 模型→中性 | 模型→负面 |
|--|-----------|-----------|-----------|
| 人工→正面 | 25 | 10 | 7 |
| 人工→中性 | 23 | 36 | 9 |
| 人工→负面 | 2 | 4 | 34 |

**误判模式分析**

1. **中性→正面（23条，最大误判）**：模型把"生日快乐""攒了300抽准备抽"等中性/轻度正面表达判为正面。边界过于宽松。
2. **正面→中性（10条）**：讨论游戏机制/剧情解读的评论，虽然语气正面但模型认为是客观描述。
3. **中性→负面（9条）**：讨论角色设定/剧情的中性评论被误判为负面（可能因为涉及"魔阴身""恨意"等负面词汇但评论本身是中立讨论）。
4. **正面→负面（7条）**：反驳引战者的评论，语气激烈但立场是维护角色的。

**待优化方向**

- [ ] 调优情感分析 prompt，明确正面/中性边界（如"单纯祝福语=中性"还是"正面"需要统一标准）
- [ ] 考虑加入"语境"提示：B站评论区的"攒抽""准备抽"是否算正面态度
- [ ] 反驳型评论的立场识别（与议题分析的 stance 标注联动）

### 2026-06-05 v3.3 情感分析性能优化

**问题**

BV1wYdWB5EVF（2933条评论）跑情感分析耗时过长：294批×(2s等待+3s API响应) ≈ 15-20分钟。

**优化措施**

| 参数 | 优化前 | 优化后 |
|------|--------|--------|
| 批大小 | 10条/请求 | 30条/请求 |
| 并发数 | 串行 | 4路线程池并发 |
| 批间等待 | 2s | 0.5s（每组并发之间） |
| 重试机制 | 无 | 3次指数退避（1s, 2s） |
| max_tokens | 1024 | 2048（适配更大批次输出） |

**实测效果**

- BV1PnV46DEP4（1385条）：47批，**67.9秒**完成（原来约15分钟，提速13倍）
- 吞吐量：~21条/分钟 → ~630条/分钟（30条×21批/分钟）

**技术实现**

- `concurrent.futures.ThreadPoolExecutor` 线程池并发
- `threading.Lock` 保护进度计数器
- 每提交一组 MAX_WORKERS 个任务后 sleep 0.5s，避免瞬间打满中转站限流
- 失败批次自动重试3次（指数退避），最终失败的标记为中性

### 2026-06-05 v3.4 新提示词 A/B 测试

**背景**

原始提示词判断标准过于笼统（"表达喜爱=正面、表达不满=负面"），导致准确率63.3%，正面Precision仅50%。尝试用更细化的游戏社区专用提示词进行对比测试。

**新提示词要点**

- 明确负面边界：直接批评、间接抱怨、社区冲突信号、负面推测/谣言
- 收窄正面范围：仅直接夸奖和对游戏角色的期待/祝福
- 扩大中性范围：社交内容、对UP主祝福、个人计划、理性讨论、社区维护
- 附带few-shot示例

**测试方法**

用 `sentiment_test_v2.py` 对 BV1PnV46DEP4 的1385条评论分别用新旧提示词跑一遍，对比差异。

**结果**

- 总差异：384条（27.7%）结果发生变化
- 变化方向：正面→中性 257条 | 负面→中性 66条 | 中性→负面 30条 | 中性→正面 23条 | 正负互转 8条
- 新提示词整体趋势：把大量社交性正面评论拉回中性（方向正确）

**人工评估（Sheet3方向性变化61条）**

- 确认判错：19条（31.1%错误率）
- 主要错误模式：
  1. 中性→正面（8条）：玩梗调侃（"我干的""张口闭眼男"）被过度解读为正面
  2. 中性→负面（7条）：理性讨论（"抽新不抽旧"、讨论体系强度）被"社区冲突信号"定义误捕
  3. 正负互转（4条）：含转折/反讽的评论判断失误

**结论**

新提示词在"正面→中性"大方向上有改善，但边界判定引入了较多新错误（方向性变化31%错误率）。暂不替换，保留测试数据供后续迭代参考。

**产出文件**

- `sentiment_test_v2.py` — A/B测试脚本（独立，不影响主流程）
- `comments_BV1PnV46DEP4_full_sentiment_v2.json` — 新提示词结果
- `sentiment_diff_BV1PnV46DEP4.xlsx` — 差异对比表（Sheet1差异评论、Sheet2统计摘要、Sheet3方向性变化+人工标注）

## 待实现：议题分类立场标注

**问题**：当前典型评论排序纯按点赞+位置，导致"反驳某观点"的高赞负面评论被选为该观点的典型评论，与议题立场不符。

**方案：分类时增加立场维度 + 报告结构调整**

1. 议题分类 prompt 增加 stance 字段：
   ```json
   {"index": 1, "topics": [{"label": "科技割裂", "stance": "提出"}]}
   {"index": 2, "topics": [{"label": "科技割裂", "stance": "反驳"}]}
   ```

2. 典型评论选取逻辑：优先取 `stance=提出/认同` 的评论

3. 报告结构调整，每个议题下分两块：
   - **典型评论**（3条）：明确提出/认同该观点的
   - **社区争议**（1-2条，可选）：高赞反驳评论，说明议题存在分歧

4. 新增议题争议度指标："37条相关，25条认同/12条反驳"——帮助运营判断该议题是否代表主流意见

**预期效果**：报告更准确反映真实舆情方向，同时保留争议信息供运营参考。
